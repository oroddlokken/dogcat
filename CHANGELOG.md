# CHANGELOG

## [Unreleased]

### Fixed

- **`test_collaborator_rebase_force_push` no longer flaky** — the test ran `git rebase -i main~0` purely as a no-op preamble before simulating the squash via `git reset --soft main`. With `HOME=/dev/null` and no `GIT_EDITOR` in the test env, the interactive rebase intermittently exited nonzero and tripped `GitRepo.git()`'s default `check=True`, failing CI at random. Removed the dead call; `reset --soft` already does the work the comment claimed (closes dogcat-651n).
- **`event_log.read(limit=...)` rejects negative limits** — previously a negative limit silently sliced off the last N events (`events[:-1]`) instead of raising. The API now raises `ValueError` for any `limit < 0`; `limit=0` continues to return an empty list (closes dogcat-3r0s).
- **`dcat web propose` rejects negative `Content-Length` headers** — `int("-1") > MAX` is False, so a negative content-length used to slip past the early body-size guard and only hit the slow-path stream cap. The middleware now returns 400 for any non-numeric or negative Content-Length (closes dogcat-3r0s).
- **Time-sensitive tests gain explicit slack against clock skew** — every `before <= obj.created_at <= after` bracket now widens by ±1s and `test_stream`'s `time.sleep(0.1) + is_alive()` check is replaced with poll-with-deadline, so NTP slew, suspend/resume, or VM time-drift between two `now()` calls can no longer flip the comparison (closes dogcat-ixyi).

### Removed

- **Unused `cli_command` decorator** — defined in `cli/_helpers.py` but never wired up. Deleted as dead code (closes dogcat-5quu).

### Changed

- **Mid-file malformed JSONL is skipped, not fatal** — `JSONLStorage._load`, `InboxStorage._load`, `EventLog.read`, and the `_save_locked` event-passthrough loop all switch from "raise on any non-last malformed line" to "log + skip + record on `_bad_lines`". A truncated record from an editor crash, a partial UTF-8 byte from an interrupted append, a non-dict line, or a future enum value no longer bricks every subsequent `dcat` invocation; `dcat doctor` surfaces the count and `dcat repair-jsonl` preserves the bytes in a sidecar before compaction drops them. The exception set on the load path is also widened to include `AttributeError` and `TypeError` so a non-dict JSONL line never escapes as a raw traceback (closes dogcat-4h0s, dogcat-64wv, dogcat-4s8b, dogcat-5tix).
- **CLI mutations enforce the same length / namespace / control-byte rules as the web form** — `validate_issue` / `validate_proposal` now apply `MAX_TITLE_LEN`, `MAX_DESC_LEN`, the `^[A-Za-z0-9_-]{1,64}$` namespace pattern, and the C0/C1 control-byte strip from `dogcat.constants` to every persisted string field (title, description, notes, owner, labels, comments, closed_reason, deleted_reason). `JSONLStorage.create`/`update`/`change_namespace` and `InboxStorage.create` route through them, so `dcat propose`, `dcat new`, `dcat update --title/--description/--namespace`, `dcat rename-namespace`, and the inbox accept path can no longer persist a 200 MB title or a `\x1b[31mevil\x1b[0m` namespace that bypassed the web form. The CLI display layer (`format_issue_brief` / `format_issue_full` / proposal brief) escapes any residual control bytes via `sanitize_for_terminal` as defense-in-depth so an older record displays as `\x1b…` rather than executing the escape sequence (closes dogcat-ad4x, dogcat-308q).
- **`dcat archive` and inbox archive hold the storage / inbox lock through the rewrite** — the read+partition+atomic-replace window previously ran without the advisory file lock, so a concurrent `dcat new` / `dcat propose` append between the read and the replace was silently overwritten. Both paths now wrap the read+rewrite in `storage._file_lock()` / `inbox._file_lock()` (closes dogcat-3ogg).
- **`change_namespace`, `rename_namespace`, `prune_tombstones`, `remove_dependencies`, and the inbox equivalents reload under the lock** — previously they mutated stale in-memory state and called `_save(_reload=False)`, so a concurrent process appending between the last `_load()` and the rewrite was silently dropped. Each method now acquires the lock, reloads from disk, re-applies the mutation against the fresh state, and saves via `_save_locked`. Worst-case scenario was the long-lived web server inbox losing every proposal submitted via the form during a CLI namespace rename. `JSONLStorage._save_locked` and `InboxStorage._save_locked` are split out of `_save` so callers can compose without re-entering the advisory lock (closes dogcat-zu0c).
- **Tombstone status is final on `close()`, `update()`, and double `delete()`** — `close()` previously flipped a tombstoned issue back to CLOSED with both `deleted_at` and `closed_at` populated; `update(--status open)` after `delete` re-opened the tombstone but left `deleted_*` set; double `delete()` overwrote the original `deleted_at`/`deleted_reason`/`deleted_by`. `close()` now refuses tombstoned issues; `update()` rejects status changes from TOMBSTONE; second `delete()` is a no-op so the original forensics are preserved (closes dogcat-4g76).
- **Issue merge driver applies status finality before the `updated_at` tiebreak** — branch A `dcat delete dc-x` at 10:00 followed by branch B `dcat update dc-x --notes` at 10:01 used to keep B (later timestamp) and silently lose the tombstone. Issues now follow the same finality rank that proposals already did (`tombstone > closed > active states > draft`); within the same rank the later `updated_at` still wins. The merge driver also parses ISO timestamps to absolute UTC time before comparing, fixing cross-timezone collaborators silently keeping older edits and `Z` vs `+00:00` for the same instant comparing unequal (closes dogcat-mro6, dogcat-623e).
- **CSRF on `dcat web propose` is now single-use, origin-checked, and shape-validated** — the previous double-submit cookie + form pair was replayable for the entire 1 h session and accepted any client-controlled cookie value. Tokens are now server-issued nonces stored on `app.state.csrf_nonces` with a 10-minute TTL, consumed on a successful POST. Cookie / form values are matched against `^[A-Za-z0-9_-]{40,128}$` before the constant-time compare. Origin / Sec-Fetch-Site is validated against the request host so a foreign-origin POST is rejected even with valid tokens. `pinned_namespaces` growth is capped at `MAX_PINNED_NAMESPACES = 100` so a replay attack on the create-namespace form cannot grow the list unbounded. `dcat web propose --host 0.0.0.0` emits a stderr warning that the CSRF + nonce defenses cap abuse but do not authenticate the submitter (closes dogcat-2icd).
- **`dcat web propose` rejects oversize POST bodies before form parsing** — a `BodySizeLimitMiddleware` rejects `Content-Length > 256 KiB` (and stream-caps chunked bodies that omit the header) with a 413, before python-multipart materializes the body. A single attacker request can no longer OOM the server with a multi-GB urlencoded `description=AAAA…`; the form's real budget is namespace + ≤500 char title + ≤50 KB description + the CSRF token, well under the cap (closes dogcat-5zjh).
- **`dcat web propose` no longer reflects attacker-controlled banner text** — the `?submitted=true&id=…&title=…` success banner used to render whatever `title` the URL carried, with no signature and no check that `id` resolved to a real proposal. The GET handler now looks the title up server-side via `inbox.get(id)` and only renders the banner when the id resolves; an untrusted `&title=…` is ignored. The pre-NFKC namespace length cap (4 × `MAX_NAMESPACE_LEN`) lands before the error template, blocking the 2× bandwidth + CPU amplification path that NFKC-normalized a multi-MB attacker namespace and echoed it back (closes dogcat-khb5, dogcat-1tbd).
- **`/openapi.json` is disabled on the web propose app** — `docs_url=None` and `redoc_url=None` previously closed `/docs` and `/redoc` but left the schema endpoint at its default path, leaking the full route + form-field schema. `openapi_url=None` closes the third route too (closes dogcat-6a5j).
- **`JSONLStorage.update()` type-checks every field before `setattr`** — `_coerce_update_value` rejects `priority=True` (bool subclass of int), `labels='bug'` (would iterate as `{'b','u','g'}` and silently break filters), `status=42` (would store an int), `metadata='hello'` (would crash later in event diffing), and `description=123`. Each field gets a focused error naming the offending key and observed type, instead of corruption + cryptic crash later (closes dogcat-3o3b).
- **Cycle detection / reachability is iterative; deep dependency chains no longer crash dcat** — `deps.detect_cycles`, `deps.would_create_cycle`, `deps.get_dependency_chain`, and `cli/_validate._detect_cycles` all converted to explicit-stack DFS. A 1001-deep chain previously hit Python's default recursion limit and bricked every `dcat dep add` / `dcat doctor` / `dcat ready` invocation; the iterative form scales to ~10k+ depth (closes dogcat-1r7h).
- **Beads importer propagates every documented field and surfaces unparseable timestamps** — `migrate_from_beads` previously dropped 10 fields per issue (`external_ref`, `design`, `acceptance`, `notes`, `parent`, `closed_by`, `updated_by`, `comments`, `closed_reason`, `metadata`). All ten now thread through `migrate_issue` 1:1 by name. `parse_datetime` no longer silently coerces a non-empty unparseable timestamp to `datetime.now()` (which broke ID stability across re-imports because `IDGenerator.generate_issue_id` hashes `title:timestamp.isoformat()`); it now raises `ValueError` and the caller increments `failed` (closes dogcat-4ue5).
- **`dcat web propose` probes every web extra at startup** — only `uvicorn` was previously guarded with the friendly install hint; a missing `jinja2` leaked a raw `ModuleNotFoundError` and a missing `python-multipart` started cleanly but 500'd on every POST. The startup loop now imports `uvicorn`, `fastapi`, `jinja2`, and `multipart`, listing every missing module in a single `dogcat[web]` install hint (closes dogcat-5n9q).
- **`parse_duration` accepts ISO8601 with a `Z` suffix and is DST-safe for naive ISO inputs** — the helper used to lowercase the input before passing to `datetime.fromisoformat`, which only accepts uppercase `Z`, so `dcat snooze 2026-04-25T00:00:00Z` was rejected even though the help text claims ISO8601 support. The relative-duration regex still matches case-insensitively (`7d` / `7D`), but ISO parsing keeps the original case and translates trailing `Z` to `+00:00`. Naive ISO inputs use a real `zoneinfo` zone instead of a fixed-offset snapshot of "now", so a snooze months out doesn't drift across a DST boundary. Duration is also capped at ~100 years so a typo like `999999999999d` no longer raises `OverflowError` inside `timedelta` and a far-future snooze cannot permanently orphan an issue from list views (closes dogcat-3x9q, dogcat-dfn9).
- **`dcat doctor --post-merge` reports a clear skip reason** — when not in a git repo or when `.dogcats` lives outside the repo (common via `.dogcatrc`), the concurrent-edit detector silently no-op'd. The skip reason now shows on stderr and in the `--json` output as `post_merge_skipped`, so the user knows nothing concurrent was checked instead of assuming all-clear (closes dogcat-40t6).
- **Git subprocess calls run under the C locale** — `git._run` (and the inline `_is_default_branch` call in `storage.py`) now overlay `LC_ALL=C` / `LANG=C` on the environment, so the substring check for "not a git repository" matches under non-English locales (French "ce n'est pas un dépôt git", German "Kein Git-Repository", etc.). Without this, auto-compaction was silently disabled outside a repo for non-English collaborators (closes dogcat-4tl1).
- **`init.defaultBranch` non-conventional values warn at compaction time** — the per-repo git config is writable by any collaborator, so setting `init.defaultBranch=feature-foo` on a feature branch caused dcat to auto-compact while on that branch and produce noisy diffs against main. The unioned set is still respected (legitimate use case for projects on `develop`/`trunk`/etc.) but a non-conventional value now logs a one-line warning so the user notices before the compaction lands (closes dogcat-2wys).
- **CLI command modules load independently** — `dogcat.cli/__init__.py` previously eagerly imported every `_cmd_*` module; one broken transitive (rich/textual ABI mismatch, watchdog wheel for the wrong Python ABI, etc.) collapsed the whole CLI including the diagnostic `dcat doctor`. Each module is now loaded via `importlib.import_module` inside its own `try/except`; failures log a one-line warning naming the module and the rest of the CLI continues to work (closes dogcat-1ge7).
- **`save_config` and `save_local_config` use atomic write-temp + fsync + replace** — the truncate-then-write pattern previously left an empty TOML on ENOSPC / kill / power-loss mid-write. `_load_toml` silently treated empty TOML as `{}`, so every configured setting (namespace, visible_namespaces, git_tracking, allow_creating_namespaces) silently disappeared. Both helpers now route through `_atomic_write_toml`. `atomic_rewrite_jsonl` also captures the existing target mode and re-applies it after the rename, so a 0644-shared `issues.jsonl` no longer demotes to 0600 (NamedTemporaryFile's default) on the first compaction (closes dogcat-1s7e, dogcat-1cfd).
- **`_persist_pinned_namespace` serializes its read-modify-write under the storage lock** — two concurrent web POSTs creating distinct new namespaces previously read the old list, both appended their own, and last-writer-wins dropped one. Wrapping the read+modify+write in `advisory_file_lock(.issues.lock)` makes both updates land. Pairs with the atomic write fix above to close the corruption side of the same race (closes dogcat-436f).
- **TOML config types are validated at load time** — `_validate_config_shape` drops a list-typed key when given a string (`visible_namespaces = "frontend"` would otherwise iterate per-character into `{'f','r','o','n','t','e','d'}` and stealthily restrict visibility to single letters), drops a string-typed key when given an int (`namespace = 42` would corrupt every issue ID under `f"{namespace}-{hash}"`), and drops a bool-typed key when given a string (`allow_creating_namespaces = "false"` was truthy under `bool(...)` and silently flipped a security-sensitive toggle on). The web propose path also switches to `is True` for the `allow_creating_namespaces` resolution as defense-in-depth (closes dogcat-4o1p, dogcat-22t5).

### Fixed

- **`validate_priority` rejects bool** — `bool` is an `int` subclass and `priority=True` slipped through the `isinstance(int)` + range check, so `storage.create(Issue(priority=True))` persisted `True` on disk. The update path (`storage._coerce_update_value`) already guarded this; `validate_priority` itself now applies the same rule so `validate_issue` and `storage.create()` reject bool consistently with the existing `update()` behaviour (closes dogcat-65fm).
- **`--limit 0` and `--limit -1` no longer silently mishandled across the CLI** — every list-style command (`ready`, `blocked`, `list`, `recently-closed`, `recently-added`, `manual`, `snoozed`, the `_list_by_status` helper) used `if final_limit:` falsy checks and `final_limit = limit_arg or limit` coalescing. `--limit 0` was treated as "skip truncation" (so it returned everything) and `--limit -1` slipped through to `issues[:-1]` which silently drops the last item. New `resolve_limit` helper coalesces with `is not None`, rejects negatives via `typer.BadParameter`, and the falsy-checks have been replaced with `is not None` (closes dogcat-26a4).
- **`detect_concurrent_edits` surfaces unloadable git refs instead of silently reporting clean** — `_load_issues_at_ref` returned `{}` when `git_helpers.show_file` failed (permission denied, missing path at the ref, git unavailable), making the post-merge detector treat both branches as having no issues at base and falsely report "no concurrent edits" while real edits went unflagged. The loader now returns `None` on failure and `detect_concurrent_edits` emits an integrity warning naming the failed ref(s) so `dcat doctor --post-merge` surfaces the gap (closes dogcat-9wj2).
- **`InboxStorage.close()` refuses tombstoned proposals** — the inbox close path had no finality check and would happily flip a tombstoned proposal back to CLOSED, overwriting `deleted_at`/`deleted_by` with `closed_at`/`closed_by` and resurrecting it from `list()` filters. Mirrors the existing `JSONLStorage.close` tombstone guard (closes dogcat-5o1m).
- **Web propose rolls back `valid_namespaces` when pinned-namespace persist fails** — when `_persist_pinned_namespace` raised after the proposal write succeeded, the namespace had already been appended to the in-process `valid_namespaces` list, so the next submit's `is_new_namespace` check returned False, the persist call was skipped entirely, and the namespace vanished on restart while looking valid in this process. The rollback removes the namespace from the in-process list before re-raising, so subsequent submits correctly re-attempt persistence (closes dogcat-30jb).
- **`resolve_partial_id("")` no longer matches every issue id** — the resolver built matches via `full_id.endswith(partial_id)`, which returns True for every full id when `partial_id == ""`. With one matching issue in the database the resolver returned that issue, so `dcat close ""` / `dcat update ""` / `dcat delete ""` (and the inbox / dep / link / comment equivalents) silently mutated the only existing record on any CLI command whose first positional arg was an id and forgot to validate it. Empty / whitespace-only partials now return `None` without scanning the id set (closes dogcat-2wzc).
- **`DCAT_LOCK_TIMEOUT_SECS=inf` no longer disables the stale-lock guard** — `float('inf') > 0` is True, so an `inf` env value sailed past the positive-value check and produced `time.monotonic() + inf = inf`, making the deadline check forever False. The remediation that the docstring advertised (raise `RuntimeError` naming the lock path so the holder can be investigated) was silently disabled. Non-finite values (`inf`, `-inf`, `nan`) and non-positive values now fall back to the 30 s default via `math.isfinite` (closes dogcat-1z5u).
- **`dcat prime --opinionated` and `dcat guide` told agents to walk away from `--manual` issues** — the "Agent Integration" section in `prime` ended with "Do NOT attempt to work on manual issues. Leave them for the user." and the "Manual Issues" section in `guide` claimed manual issues "can't be handled by an AI agent". In interactive sessions, agents (notably Opus 4.7) read these literally and refused HITL work entirely. Both passages now frame `--manual` as human-in-the-loop, not "agent skips": the `--agent-only` filter exists so autonomous-batch runs (no human present) move past these, but in an interactive session the agent still drives the ticket and uses the user as hands for the steps it can't reach (closes dogcat-48sb)
- **Web propose POST no longer poisons `valid_namespaces` on a failed write** — `valid_namespaces.append(namespace)` ran before the `asyncio.to_thread(_create_proposal_sync, ...)` write. If the write failed, the namespace stayed in `valid_namespaces` for the rest of the process lifetime, so subsequent submits to that namespace bypassed the `is_new_namespace` branch and never made it into `pinned_namespaces`. Append + persist now happen only after a successful write (closes dogcat-39hi)
- **Compaction no longer silently drops malformed event records** — `_save_locked` re-reads the source file to preserve event records and previously skipped parse failures with `continue`, so corrupt events vanished on routine compaction with no log. Each malformed line now logs a warning naming the line number and file so the user can recover from git (closes dogcat-igrw)
- **`prune_tombstones` no longer leaks a file handle** — the JSONL re-read in `prune_tombstones` opened the file via `for raw_line in self.path.open("rb"):` without a context manager. Wrapped in `with self.path.open("rb") as src:` matching every other read site (closes dogcat-1qcb)
- **Web `allow_creating_namespaces` `getattr` defaults flipped to False** — both call sites in `web/propose/routes.py` defaulted to `True` when the attribute was missing on `app.state`, the opposite of `create_app`'s default of False — so a bare app construction silently allowed namespace minting. Defaults are now False, matching the factory (closes dogcat-kz1a)
- **Inbox accept rolls back the local issue when remote close fails** — `dcat inbox accept` previously called `storage.create(issue)` and then `remote_inbox.close(...)`; if close failed (network/lock/OSError), the local issue lingered with no record on the remote that the proposal had been accepted. The accept flow now tombstones the local issue when remote close raises, restoring the pre-accept state and surfacing the failure (closes dogcat-2k1s)
- **`typing_extensions` declared as a runtime dependency** — `models.py` imports `Self` from `typing_extensions` (still required for Python 3.10), but the package was only present transitively via textual. Adding `typing_extensions>=4.0.0` to `pyproject.toml` removes the implicit dependency on textual's transitive closure (closes dogcat-2ymd)
- **`storage.check_id_uniqueness` now actually checks** — the function unconditionally returned True, making the doctor's "All issue IDs are unique" check meaningless. It now scans the JSONL log for issue records sharing a `full_id` with differing `created_at` values — the signature of two distinct issues collapsing onto the same hash via concurrent merges or hand-edits (closes dogcat-13og)
- **`_is_default_branch` no longer treats permission errors as 'safe to compact'** — any non-zero git rc fell through to "not a git repo, compact freely". - **`config.toml` parse failures now log a warning** — `_load_toml` swallowed `TOMLDecodeError` and returned `{}`, so a typo in `config.toml` produced "all defaults" with no signal. Parse errors now log a warning at WARNING level naming the path and the parse error (closes dogcat-5ctk)

### Security

- **CSRF tokens on `dcat web propose` are single-use, origin-bound, and shape-validated** — closes a 1-hour replay window where a captured `(cookie, form-token)` pair was reusable from any HTTP client. Defenses: server-issued nonce table with 10-minute TTL consumed on POST, Origin / Sec-Fetch-Site allowlist matching the request host, `^[A-Za-z0-9_-]{40,128}$` shape check on cookie + form values before the constant-time compare, and a `MAX_PINNED_NAMESPACES = 100` cap so a replay attack on the create-namespace form cannot grow `pinned_namespaces` unbounded. `dcat web propose --host 0.0.0.0` emits a stderr warning that CSRF + nonces cap abuse but do not authenticate the submitter (closes dogcat-2icd).
- **Body-size middleware blocks DoS via giant POST bodies on `dcat web propose`** — without the guard, python-multipart fully buffered the body before any route field check ran, so a single attacker request with `Content-Length: 10737418240` (10 GB) of urlencoded `description=AAAA…` could OOM the server. `BodySizeLimitMiddleware` rejects `Content-Length > 256 KiB` early and stream-caps chunked bodies that omit the header, returning 413 before materialization. The form's real budget is well under the cap (closes dogcat-5zjh).
- **Stripped C0/C1 control bytes from every persisted string field** — `validate_issue` / `validate_proposal` strip `\x00–\x1f` (except `\t`/`\n`), `\x7f`, and `\x80–\x9f` from title, description, notes, owner, labels, comments, `closed_reason`, and `deleted_reason`. A web POST `title='\x1b[2J\x1b[H\x1b[31mFAKE PROMPT'` could previously clear a maintainer's terminal on `dcat inbox list` and overwrite it with a forged shell prompt; for agents-in-the-loop, an LLM consuming inbox titles via dcat would have been prompt-injected. The CLI display layer (`format_issue_brief` / `format_issue_full`) escapes any residual control bytes via `sanitize_for_terminal` as defense-in-depth (closes dogcat-308q).
- **Closed the namespace bandwidth + CPU amplification path on `dcat web propose`** — the form normalized via NFKC and rendered the unbounded user-supplied namespace into the error template before the length check. `ns_size=5_000_000` yielded `resp_len=10_003_680` (2× amplification) plus significant CPU on NFKC over the full payload. The route now pre-caps the raw namespace at `4 × MAX_NAMESPACE_LEN` before normalization or rendering, and the rejection path substitutes a placeholder so a multi-MB attacker namespace never reaches the template (closes dogcat-1tbd).
- **Closed the schema-disclosure surface on `dcat web propose`** — `docs_url=None` and `redoc_url=None` previously closed `/docs` and `/redoc` but left `/openapi.json` open at its default path, leaking the full route + form-field schema to anyone on the bind address (LAN if `--host 0.0.0.0`). `openapi_url=None` closes that route too (closes dogcat-6a5j).
- **Closed the reflected banner phishing surface on `dcat web propose`** — `?submitted=true&id=…&title=…` rendered an attacker-controlled "success" banner inside dogcat's brand chrome with the real CSRF cookie attached. The handler now looks the title up server-side via `inbox.get(id)` and only renders the banner when the id resolves to a live proposal, ignoring any `&title=…` in the query (closes dogcat-khb5).
- **Bounded `.dogcatrc` walk-up to the git toplevel / `$HOME`** — the previous walk reached the filesystem root, taking the first `.dogcatrc` found, with no ownership check, no boundary, no warning. Anyone with write access to `/tmp/.dogcatrc` (multi-tenant host, shared mount, CI workspace) could silently re-root every dcat command in that subtree to attacker-controlled storage. Issues created by the victim landed in the attacker's tree, attributed to the victim's git email. Walk-up now stops at the git toplevel (or `$HOME` outside a repo); an out-of-tree rc target emits a stderr warning; rc files whose owner differs from the target's owner are refused unless `DCAT_UNSAFE_CROSS_USER=1` is set. Set `DCAT_RC_WALKUP_UNRESTRICTED=1` to opt back into the legacy behavior (closes dogcat-4107).
- **`init.defaultBranch` set to a non-conventional name now warns at compaction time** — the per-repo git config is writable by any collaborator, so setting `init.defaultBranch=feature-foo` carried into a feature branch caused dcat to auto-compact while on that branch and produce noisy, conflict-prone diffs against main. The unioned set is still respected (legitimate use case for `develop`/`trunk`/etc.) but a non-conventional value emits a one-line warning recommending the user move the override to project config where review is more visible (closes dogcat-2wys).
- **Per-session CSRF tokens on `dcat web propose`** — the previous token was generated once at app startup via `secrets.token_urlsafe(32)` and shared across every browser session, so any client that ever obtained it could reuse it indefinitely until restart. Replaced with a double-submit cookie (`dcat_csrf`, HttpOnly, SameSite=Strict, 1h max-age): each session gets its own token, and tokens expire (closes dogcat-5dd4)
- **Whitelisted namespace format on the web proposal form** — the form previously accepted arbitrary strings for `namespace`, including spaces, control characters, and Unicode homoglyphs that could spoof an existing namespace visually. Input is now NFKC-normalized then matched against `^[A-Za-z0-9_-]{1,64}$`; fullwidth ASCII folds to plain ASCII before the check, but Cyrillic look-alikes (and similar) are rejected (closes dogcat-1819)
